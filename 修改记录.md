# 修改记录

## 2026-02-27 彻底优化存储性能（目标 100ms）

### 根本原因分析

**问题**：存储很慢，一秒后都没有存储上。

**根本原因**：每次 IndexedDB 操作后都调用 `scheduleUtoolsBackup()`！

```typescript
// 问题代码：每次操作都触发备份
await db.wordRecords.add(wordRecord)
scheduleUtoolsBackup()  // ← 触发整个数据库导出！

await db.wordProgress.put(progress)
scheduleUtoolsBackup()  // ← 又触发一次！

await db.dailyRecords.put(record)
scheduleUtoolsBackup()  // ← 再触发一次！
```

**`scheduleUtoolsBackup()` 做了什么**：
1. 动态导入 `pako` 和 `dexie-export-import`
2. 导出整个 IndexedDB 数据库
3. 转换为 JSON 文本
4. gzip 压缩
5. 写入 uTools DB

这个过程可能需要 **数百毫秒到数秒**！

### 优化方案

**1. 移除每次操作后的 `scheduleUtoolsBackup()` 调用**

只在以下时机备份：
- 学习会话结束时（`useSaveLearningRecord`）
- 页面退出时（`beforeunload`、`visibilitychange`）
- 定时备份（30秒间隔）

**2. 存储操作完全异步，不阻塞 UI**

```typescript
// 优化后：立即进入下一个单词，存储在后台完成
saveWordRecord(...).catch(...)
updateWordProgress(...).then(...).catch(...)
onFinish()  // 立即调用，不等待
```

**3. 添加性能日志**

```typescript
console.log(`[DB] saveWordRecord done in ${performance.now() - startTime}ms`)
console.log(`[DB] updateWordProgress done in ${performance.now() - startTime}ms`)
console.log(`[DB] All IndexedDB operations done in ${performance.now() - startTime}ms`)
```

### 修改文件

| 文件 | 修改内容 |
|------|---------|
| `src/utils/db/index.ts` | 移除 `saveWordRecord` 中的 `scheduleUtoolsBackup()` |
| `src/utils/db/hooks/useWordProgress.ts` | 移除 `updateWordProgress` 中的 `scheduleUtoolsBackup()` |
| `src/utils/db/hooks/useDailyRecord.ts` | 移除 `incrementXxx` 中的 `scheduleUtoolsBackup()` |
| `src/pages/Typing/.../useWordCompletion.ts` | 异步存储，立即调用 `onFinish()` |

### 性能对比

| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| 单词存储 | 等待所有操作完成（可能 1s+） | 立即进入下一个单词 |
| uTools 备份 | 每次操作都触发 | 仅学习结束时触发 |
| 用户感知延迟 | 明显卡顿 | 无感知 |

### 预期效果

- **IndexedDB 操作**：应该在 **10-50ms** 内完成
- **UI 响应**：立即进入下一个单词
- **uTools 备份**：后台异步执行，不影响用户体验

## 2026-02-27 优化存储效率

### 问题分析

用户反馈：存储操作很慢，快速完成单词后退出数据丢失。

**效率问题**：每个单词完成时有 4-5 次串行数据库操作：

```
saveWordRecord()        → 写入 wordRecords 表（等待）
    ↓
updateWordProgress()    → 读取 wordProgress → 写入 wordProgress（等待）
    ↓
incrementLearned()      → 事务：读取 dailyRecords → 写入 dailyRecords（等待）
```

### 优化方案

**1. 使用 `put` 替代 `update`**

Dexie 的 `put` 方法会自动判断插入或更新，减少一次查询：

```typescript
// 优化前：需要先查询 id，再 update
if (progress.id) {
  await db.wordProgress.update(progress.id, progress)
} else {
  progress.id = await db.wordProgress.add(progress)
}

// 优化后：put 自动判断
progress.id = await db.wordProgress.put(progress)
```

**2. 并行执行独立操作**

`saveWordRecord` 和 `updateWordProgress` 是独立的，可以并行执行：

```typescript
// 优化前：串行执行
await saveWordRecord(...)
await updateWordProgress(...)

// 优化后：并行执行
const [, progress] = await Promise.all([
  saveWordRecord(...),
  updateWordProgress(...),
])
```

**3. 简化事务操作**

在事务内直接使用索引查询，避免调用外部函数：

```typescript
// 优化前：调用外部 getTodayRecord()
const record = await getTodayRecord()

// 优化后：事务内直接查询
const today = getTodayDate()
let record = await db.dailyRecords.where('[dict+date]').equals([resolvedDictId, today]).first()
```

### 修改文件

- `src/utils/db/hooks/useWordProgress.ts`：使用 `put` 替代 `update`
- `src/utils/db/hooks/useDailyRecord.ts`：简化事务操作
- `src/pages/Typing/components/WordPanel/components/Word/hooks/useWordCompletion.ts`：并行执行独立操作

### 性能对比

| 操作 | 优化前 | 优化后 |
|------|--------|--------|
| saveWordRecord | 串行 | 并行 |
| updateWordProgress | 2次DB操作（查询+更新） | 1次DB操作（put） |
| incrementLearned | 事务+外部函数调用 | 事务内直接查询 |

预计每个单词的存储时间减少 **50% 以上**。

## 2026-02-27 修复快速退出导致数据丢失问题

### 问题分析

用户反馈：快速完成一个单词后立即退出，数据不会记录；但如果等待一段时间就是好的。

**根本原因**：
1. `useWordCompletion` 中 `saveAllRecords()` 是异步的，但 `onFinish()` 在它之后立即调用
2. `onFinish()` 会触发 `NEXT_WORD` 或 `FINISH_WORDS`，可能导致组件卸载
3. 组件卸载后，异步操作被中断，IndexedDB 写入未完成

**存储逻辑原则**：
- **IndexedDB 操作**：必须立即执行并等待完成
- **uTools DB 备份**：可以延迟（1.5秒防抖是合理的）

### 修改内容

**文件**: `src/pages/Typing/components/WordPanel/components/Word/hooks/useWordCompletion.ts`

修改前：
```typescript
saveAllRecords()
onFinish()  // 立即调用，不等待保存完成
```

修改后：
```typescript
saveAllRecords().then(() => {
  onFinish()  // 等待 IndexedDB 操作完成后再进入下一个单词
})
```

### 修改结果

- 单词记录、进度更新、每日统计等 IndexedDB 操作完成后才进入下一个单词
- 用户快速退出时数据不会丢失
- uTools DB 备份仍然使用防抖机制，不影响性能

  - 添加 `await` 等待 `saveLearningRecord` 完成

**3. 添加事务保护到 useDailyRecord**

- `src/utils/db/hooks/useDailyRecord.ts`：
  - 使用 Dexie 事务包装数据库操作
  - 在事务内完成读取、更新、写入操作
  - 事务成功后再更新 UI 状态
  - 添加错误处理

**4. 修复 isSavingRecord 状态管理问题**

- `src/utils/db/index.ts`：
  - `useSaveWordRecord` 返回 `Promise<number>`，不再内部设置 `isSavingRecord`
  - 移除内部的 `SET_IS_SAVING_RECORD` 派发

  - 将所有异步操作包装在 `saveAllRecords` 函数中
  - 等待所有操作完成后再设置 `isSavingRecord = false`
  - 使用 `try-finally` 确保状态正确重置
  - **关键修复**：`saveAllRecords().then(() => onFinish())`，等待 IndexedDB 操作完成后再进入下一个单词

### 修改结果

- 备份操作统一由 `scheduleUtoolsBackup` 管理，避免重复
- 所有异步操作正确等待完成
- 数据库操作使用事务保护，确保一致性
- 状态管理正确反映保存进度
- 错误处理更完善

## 2026-02-27 修复学习记录丢失问题

### 问题分析

用户反馈学习记录第二天会丢失。

**根本原因**：应用使用两层存储机制：
1. **IndexedDB**（Dexie 库，数据库名 `RecordDB`）- 运行时数据存储
2. **uTools.db**（uTools 提供的持久化存储）- 备份数据存储

数据流问题：
- 保存数据：IndexedDB → uTools.db ✅ 已实现
- 恢复数据：uTools.db → IndexedDB ❌ **缺失！**

应用启动时没有调用 `importDatabase2UTools()` 来恢复数据，导致 IndexedDB 被清理后数据丢失。

### 修改内容

**文件**: `src/index.tsx`

1. **添加数据恢复状态**：
   - 添加 `isDataRestored` state，用于跟踪数据恢复是否完成

2. **添加数据恢复逻辑**：
   - 在应用启动时检查 uTools.db 中是否有备份数据
   - 如果有数据，调用 `window.importDatabase2UTools()` 恢复到 IndexedDB
   - 恢复完成后再渲染页面

3. **更新渲染条件**：
   - 将渲染条件从 `!isModeReady` 改为 `!isModeReady || !isDataRestored`
   - 确保数据恢复完成后再显示页面

### 修改结果

- 应用启动时自动从 uTools.db 恢复学习记录
- 重启电脑、清除浏览器缓存后数据不再丢失
- 数据持久化流程完整：保存时备份到 uTools.db，启动时从 uTools.db 恢复

## 2026-02-26 修复单词加载竞态条件问题

```

  "wordListLength": 104,
  "wordListType": "object",
  "wordListIsArray": true,
```

**问题**：`isLoadingRef` 显示为 `false`，但还是被跳过了！

**根本原因**：使用 `useRef` 管理加载状态存在竞态条件。`useEffect` 依赖数组变化时，`loadLearningWords` 被重新创建，但 `isLoadingRef` 的闭包捕获了过时的值。

### 修改内容

**文件**: `src/pages/Typing/hooks/useWordList.ts`

1. **移除 `useRef` 加载状态**：
   - 删除 `const isLoadingRef = useRef(false)`
   - 改为使用 `useState`：`const [isLoadingLearningWords, setIsLoadingLearningWords] = useState(false)`

2. **更新加载状态检查**：
   - `if (isLoadingRef.current)` → `if (isLoadingLearningWords)`
   - `isLoadingRef.current = true` → `setIsLoadingLearningWords(true)`
   - `isLoadingRef.current = false` → `setIsLoadingLearningWords(false)`

3. **更新依赖数组**：
   - 在 `loadLearningWords` 的 `useCallback` 依赖数组中添加 `isLoadingLearningWords`

### 修改结果

- 使用 `useState` 替代 `useRef` 管理加载状态，确保状态变化触发重新渲染

## 2026-02-18 修复释义被状态重置覆盖的问题

### 修改目的
修复 MDX 查询成功后释义被状态重置覆盖的问题。

### 问题分析

通过调试日志发现：

```
[WordPanel] Rendering - trans: Array(1)  // 释义获取成功
[WordPanel] Rendering - trans: Array(0)  // 释义又没了
```

**根本原因**：

1. 初始渲染时 `dailyRecord` 为 `null`
2. MDX 异步查询成功，释义被添加到单词数据
3. `dailyRecord` 异步加载完成，从 `null` 变为有值
4. 触发 `reloadWords()` → `loadLearningWords()` → `learningWords` 重新计算
5. `words` 引用变化，触发 `SETUP_CHAPTER`
6. **状态被重置**，释义丢失！

### 修改内容

**文件**: `src/pages/Typing/store/index.ts`

修改 `SETUP_CHAPTER` action，在设置新单词时保留已有单词的释义信息：

```typescript
case TypingStateActionType.SETUP_CHAPTER: {
  const newState = structuredClone(initialState)
  const newWords = action.payload.shouldShuffle ? shuffle(action.payload.words) : action.payload.words
  
  // 保留已有单词的释义信息
  const oldWordsMap = new Map(state.chapterData.words.map(w => [w.name, w]))
  newState.chapterData.words = newWords.map(w => {
    const oldWord = oldWordsMap.get(w.name)
    if (oldWord && oldWord.trans && oldWord.trans.length > 0) {
      return { ...w, trans: oldWord.trans, ukphone: oldWord.ukphone || w.ukphone, tense: oldWord.tense || w.tense }
    }
    return w
  })
  
  newState.isTyping = true
  newState.isTransVisible = state.isTransVisible
  newState.isImmersiveMode = state.isImmersiveMode
  return newState
}
```

### 修改结果
- MDX 查询成功后，释义不会被状态重置覆盖
- 单词释义、音标、时态信息在状态重置时得以保留

## 2026-02-19 移除 SETUP_CHAPTER，用 SET_WORDS 替代

### 修改目的
移除过时的 `SETUP_CHAPTER` action，用更精确的 `SET_WORDS` 和 `RESET_PROGRESS` 替代，从根本上解决状态重置问题。

### 问题分析

#### `SETUP_CHAPTER` 的问题

| 问题 | 说明 |
|------|------|
| **命名过时** | 现在没有"章节"概念，用户是连续背单词 |
| **语义混乱** | 既设置单词列表，又重置 index，又重置 UI 状态 |
| **触发时机不当** | 在 `words` 变化时自动触发，但用户可能正在背单词 |
| **职责不清** | 做了太多事情，难以精确控制 |

#### 重构前的问题流程

```
用户正在背第2个单词 (index = 1)
    │
    ▼
useWordList 重新计算
    │
    ▼
words 引用变化（但内容相同）
    │
    ▼
SETUP_CHAPTER 触发
    │
    ▼
index 被重置为 0
    │
    ▼
用户被踢回第1个单词！
```

### 重构方案

```
重构前：
┌─────────────────────────────────────────────────────────────────────┐
│  SETUP_CHAPTER                                                       │
│    - 设置单词列表                                                    │
│    - 重置 index = 0                                                  │
│    - 重置 UI 状态                                                    │
│    - 可选打乱顺序                                                    │
│                                                                      │
│  问题：一个 action 做太多事，无法精确控制                            │
└─────────────────────────────────────────────────────────────────────┘

重构后：
┌─────────────────────────────────────────────────────────────────────┐
│  SET_WORDS                                                           │
│    - 只设置单词列表                                                  │
│    - 尝试保持当前单词位置                                            │
│    - 不重置统计数据                                                  │
│                                                                      │
│  RESET_PROGRESS                                                      │
│    - 用户主动重置时使用                                              │
│    - 重置 index = 0                                                  │
│    - 重置 UI 状态                                                    │
│                                                                      │
│  优点：职责分离，精确控制                                            │
└─────────────────────────────────────────────────────────────────────┘
```

### 修改内容

**文件**: `src/pages/Typing/store/index.ts`

```typescript
// 新增 SET_WORDS action
case TypingStateActionType.SET_WORDS: {
  const newWords = action.payload.shouldShuffle ? shuffle(action.payload.words) : action.payload.words
  // 尝试保持当前单词位置
  const currentWordName = state.chapterData.words[state.chapterData.index]?.name
  const newIndex = currentWordName 
    ? newWords.findIndex(w => w.name === currentWordName)
    : 0
  
  return {
    ...state,
    chapterData: {
      words: newWords,
      index: newIndex >= 0 ? newIndex : 0,  // 保持位置，而不是重置为 0
    },
    uiState: {
      ...state.uiState,
      isTyping: true,
    },
  }
}

// 新增 RESET_PROGRESS action（用户主动重置时使用）
case TypingStateActionType.RESET_PROGRESS: {
  return {
    ...state,
    chapterData: {
      ...state.chapterData,
      index: 0,
    },
    uiState: {
      ...initialUIState,
      isTyping: true,
    },
  }
}
```

## 2026-02-16 修复单词顺序与发音顺序不一致问题

### 修改目的
修复背单词时单词被跳过的问题，确保每个单词都能正确显示和播放发音。

### 问题分析
1. **React StrictMode 导致 onFinish 被调用两次**：在 `WordComponent` 中，当单词输入完成时，`onFinish()` 在 useEffect 中被调用。由于应用使用了 `React.StrictMode`，开发环境下 useEffect 会执行两次，导致 `NEXT_WORD` 被连续调用两次，跳过了中间的单词。

2. **播放标记未重置**：在 `WordSound` 组件中，`hasPlayedRef.current` 用于标记当前单词是否已播放发音，但单词切换时没有重置。

### 修改内容

#### 1. 防止 onFinish 重复调用
**文件**: `src/pages/Typing/components/WordPanel/components/Word/index.tsx`

使用 ref 标记是否已调用 onFinish，防止 React StrictMode 导致的重复调用：

```typescript
// 新增 ref
const onFinishCalledRef = useRef(false)

// 单词变化时重置标记
useEffect(() => {
  // ...
  onFinishCalledRef.current = false
}, [word, setWordState])

// 防止重复调用
useEffect(() => {
  if (wordState.isFinished) {
    if (onFinishCalledRef.current) return
    onFinishCalledRef.current = true
    // ...
    onFinish()
  }
}, [...])
```

#### 2. 重置播放标记
**文件**: `src/pages/Typing/components/WordPanel/components/WordSound/index.tsx`

在单词变化时重置 `hasPlayedRef.current`：

```typescript
useEffect(() => {
  hasPlayedRef.current = false
  return () => stopRef.current()
}, [word])
```

#### 3. 预加载当前单词发音
**文件**: `src/pages/Typing/components/WordPanel/index.tsx`

同时预加载当前单词和下一个单词的发音：

```typescript
usePrefetchPronunciationSound(currentWord?.name)
usePrefetchPronunciationSound(nextWord?.name)
```

### 修改结果
- 单词不会被跳过，每个单词都能正确显示
- 发音与显示的单词一致
- 当前单词和下一个单词的发音都会被预加载

## 2026-02-14 导入词典逻辑说明

### 功能概述
用户可以通过拖拽或选择 Excel 文件导入自定义词典。

### 导入流程

```
用户选择 Excel 文件
         ↓
┌─────────────────────────────────────┐
│  1. 文件验证                          │
│  - 检查文件类型 (xls/xlsx/json)       │
│  - 检查文件大小 (≤2MB)                │
└─────────────────────────────────────┘
         ↓
┌─────────────────────────────────────┐
│  2. Excel 转 JSON                     │
│  - 使用 XLSX 库解析 Excel             │
│  - 验证表头必须有 name 和 trans 列    │
│  - 转换为标准 JSON 格式               │
└─────────────────────────────────────┘
         ↓
┌─────────────────────────────────────┐
│  3. 创建词典元数据                    │
│  - 生成唯一 ID (x-dict-{uuid})       │
│  - 计算章节数量 (每章20词)            │
└─────────────────────────────────────┘
         ↓
┌─────────────────────────────────────┐
│  4. 存储到 uTools 数据库              │
│  - 词典内容: utools.db.put({_id, data})│
│  - 配置列表: local-dict-config        │
└─────────────────────────────────────┘
```

### 涉及文件

| 文件 | 功能 |
|------|------|
| `Form4AddDict/index.tsx` | 表单界面、文件选择、验证 |
| `convertExcelToJson.ts` | Excel 转 JSON 格式 |
| `preload.js` | uTools 数据库存储操作 |

### Excel 格式要求

| 列名 | 必填 | 说明 |
|------|------|------|
| name | ✅ | 单词 |
| trans | ✅ | 翻译 |
| usphone | ❌ | 美式音标 |
| ukphone | ❌ | 英式音标 |
| notation | ❌ | 注解 |

### JSON 输出格式

```json
[
  {
    "name": "apple",
    "trans": ["苹果"],
    "usphone": "'æpl",
    "ukphone": "'æpl"
  }
]
```

### 存储结构

```
uTools 数据库
├── local-dict-config     # 词典配置列表
│   └── [{id, name, length, chapterCount, ...}, ...]
│
├── x-dict-{uuid-1}       # 词典1内容
│   └── [{name, trans, usphone, ukphone}, ...]
│
└── x-dict-{uuid-2}       # 词典2内容
    └── [{name, trans, usphone, ukphone}, ...]
```

### 关键函数

#### `convertExcelToJson(file: File)`
- 使用 XLSX 库读取 Excel 文件
- 验证表头格式
- 转换为 JSON 数组

#### `saveDict(formData, jsonData)`
- 创建词典元数据 (id, name, length, chapterCount 等)
- 调用 `window.newLocalDictFromJson` 保存

#### `window.newLocalDictFromJson(jsonData, dictMeta)`
- 保存词典内容到 uTools 数据库
- 更新词典配置列表
- 检查是否已存在，避免重复添加

3. **Invert**: 计算位置差值，用 `transform: translateY()` 反向偏移
4. **Play**: 移除偏移，让 CSS transition 播放动画

**代码实现**:
```typescript

const recordPositions = useCallback(() => {
  const items = listRef.current.querySelectorAll('.dict-item')
  items.forEach((item) => {
    const path = item.getAttribute('data-path')
    if (path) {
      prevPositions.current.set(path, item.getBoundingClientRect())
    }
  })
}, [])

// 播放 FLIP 动画
const playFlipAnimation = useCallback(() => {
  const items = listRef.current.querySelectorAll('.dict-item')
  items.forEach((item) => {
    const prevRect = prevPositions.current.get(path)
    if (!prevRect) return
    
    const currentRect = item.getBoundingClientRect()
    const deltaY = prevRect.top - currentRect.top
    
    if (deltaY !== 0) {
      item.style.transform = `translateY(${deltaY}px)`
      item.style.transition = 'none'
      
      requestAnimationFrame(() => {
        item.style.transition = 'transform 0.3s ease'
        item.style.transform = 'translateY(0)'
      })
    }
  })
}, [])
```

### 修改结果
- 点击上移/下移按钮时，词典项平滑过渡到新位置

### 修改目的

#### 问题1：play() 函数未停止当前播放
- `usePronunciation.ts` 中的 `play()` 函数直接调用 `sound.play()`

- 导致 useEffect 依赖变化，触发额外的播放
